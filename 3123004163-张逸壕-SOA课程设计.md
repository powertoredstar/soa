# 概论

## 选题以及意义

即时通讯（Instant Messaging）是互联网最基础的应用之一，从早期的 IRC、MSN 到如今的微信、QQ，即时聊天软件已经成为人们日常沟通不可或缺的工具。本项目选择"Java 即时聊天软件设计与实现"作为课程设计题目，旨在通过设计并实现一个基于 Socket 的多用户聊天系统，深入理解网络编程、多线程并发、消息队列等核心技术。

该软件适用于小型局域网或本机环境下的多人即时通讯场景，可以实现以下功能：

- **用户管理**：注册新用户、登录/退出、查看在线用户列表
- **私聊通信**：在线用户之间可以发送和接收私聊消息
- **数据持久化**：用户注册信息保存到本地 JSON 文件，服务重启后不丢失

通过本项目，用户可以在命令行终端（telnet）上完成注册、登录、查找在线用户、发送私聊消息等操作，解决多用户之间实时文字通信的需求。

## 相关技术介绍

### Socket 网络编程

Socket（套接字）是网络通信的端点，提供进程间通信的机制。本项目使用 TCP 协议的 Socket 通信，确保消息的可靠传输。

- **服务端**：`ServerSocket` 绑定端口，调用 `accept()` 等待客户端连接
- **客户端**：使用 `telnet` 命令连接服务器端口，发送文本命令进行交互

### 多线程与线程池

为支持多个客户端同时连接，服务器采用多线程模型：

- 每个客户端连接由一个独立的线程处理（`ClientHandler`）
- 使用 `ThreadPoolExecutor` 线程池管理并发连接，避免无限制创建线程
- 通过 `threading.Lock` 保护共享数据（在线用户映射表）的线程安全

### 消息队列（生产者-消费者模式）

私聊消息的转发采用消息队列实现解耦：

- **生产者**：`ClientHandler` 接收到私聊请求后，将消息放入 `queue.Queue`
- **消费者**：`SendMessageService` 线程持续从队列中取出消息，查找目标用户并转发
- 优势：发送方和接收方解耦，支持消息缓存，避免阻塞

### 数据持久化

用户注册信息采用 JSON 文件存储，通过 `FileUserStore` 类实现读写操作，保留升级到数据库的接口。

## 用例分析

### 用例1：用户注册

- **参与者**：未注册用户
- **前置条件**：服务器已启动，用户通过 telnet 连接到服务器
- **主流程**：
  1. 用户输入命令 `001,用户名,密码`
  2. 服务器检查用户名是否已存在
  3. 若不存在，创建新用户并保存到文件
  4. 返回注册结果 `101,用户名,OK` 或 `101,用户名,FAIL:原因`
- **后置条件**：新用户信息已持久化保存

### 用例2：用户登录

- **参与者**：已注册用户
- **前置条件**：用户已注册，当前未登录
- **主流程**：
  1. 用户输入命令 `002,用户名,密码`
  2. 服务器验证用户名和密码
  3. 验证通过后将用户加入在线列表
  4. 返回登录结果 `102,用户名,OK`
- **后置条件**：用户出现在在线用户列表中，可使用聊天功能

### 用例3：发送私聊消息

- **参与者**：已登录用户（发送方）
- **前置条件**：发送方已登录
- **主流程**：
  1. 发送方输入命令 `005,自己,对方,消息内容`
  2. 服务器校验发送方已登录且身份匹配
  3. 将消息放入消息队列
  4. 返回确认 `ACK,对方,OK`
  5. 消息转发线程取出消息，查找目标用户
  6. 若目标在线，转发消息 `106,发送方,接收方,消息内容`
  7. 若目标不在线，返回错误 `ERR,对方,FAIL:目标用户不在线`
- **后置条件**：接收方收到消息或发送方收到错误提示

### 用例4：查看在线用户

- **参与者**：已登录用户
- **前置条件**：用户已登录
- **主流程**：
  1. 用户输入命令 `004,all`
  2. 服务器返回在线用户列表 `104,请求者,用户1,用户2,...`
- **后置条件**：无

### 用例5：用户退出

- **参与者**：已登录用户
- **前置条件**：用户已登录
- **主流程**：
  1. 用户输入命令 `003,用户名`
  2. 服务器将用户从在线列表移除
  3. 返回退出确认 `103,用户名`
- **后置条件**：用户从在线列表移除，连接关闭

# 总体设计

## 系统架构

系统采用 C/S（客户端/服务器）架构，分为以下模块：

```
┌─────────────┐         ┌──────────────────────────────────────────┐
│   Client 1  │◄──────►│                                          │
│  (telnet)   │         │           Chat Server                    │
├─────────────┤         │  ┌─────────────┐  ┌──────────────────┐  │
│   Client 2  │◄──────►│  │ Thread Pool  │  │ SendMessageSvc   │  │
│  (telnet)   │         │  │ (30 workers) │  │ (consumer thread)│  │
├─────────────┤         │  └──────┬───────┘  └────────▲─────────┘  │
│   Client 3  │◄──────►│         │                    │            │
│  (telnet)   │         │  ┌──────▼───────┐  ┌───────┴───────┐    │
└─────────────┘         │  │ClientHandler │  │ Message Queue │    │
                        │  │ (per client) │──►│ (queue.Queue) │    │
                        │  └──────┬───────┘  └───────────────┘    │
                        │         │                                │
                        │  ┌──────▼───────┐  ┌───────────────┐    │
                        │  │  OnlineUsers │  │  FileUserStore│    │
                        │  │ (在线用户表)  │  │ (用户持久化)   │    │
                        │  └──────────────┘  └───────┬───────┘    │
                        │                             │            │
                        │                    ┌───────▼───────┐    │
                        │                    │  users.json   │    │
                        │                    │ (数据文件)     │    │
                        │                    └───────────────┘    │
                        └──────────────────────────────────────────┘
```

## 模块划分

| 模块 | 职责 |
|------|------|
| `ChatServer` | 主服务器，绑定端口，接受连接，分配线程 |
| `ClientHandler` | 处理单个客户端的命令解析与响应 |
| `SendMessageService` | 消息转发线程，从队列取出消息并投递 |
| `OnlineUsers` | 在线用户管理（用户名→Socket 映射），线程安全 |
| `FileUserStore` | 用户注册信息的 JSON 文件读写 |
| `UserRegistry` | 内存中的用户注册数据（用户名→密码） |

# 详细设计

## ChatServer 主服务器

主服务器负责启动各项服务并接受客户端连接：

1. 初始化 `FileUserStore`，加载 `data/users.json` 中的用户数据
2. 创建 `queue.Queue` 消息队列和 `OnlineUsers` 在线用户管理器
3. 启动 `SendMessageService` 消息转发线程
4. 绑定 `localhost:12345`，循环调用 `accept()` 接受连接
5. 每个新连接通过 `ThreadPoolExecutor(max_workers=30)` 提交到线程池处理

## ClientHandler 客户端处理器

每个客户端连接由一个 `ClientHandler` 处理，核心逻辑为命令循环：

1. 连接建立后发送欢迎信息（包含学号 `3123004163`）和命令帮助
2. 循环读取客户端发送的一行文本
3. 按逗号分隔解析命令码和数据部分
4. 根据命令码分发处理：
   - `001`：调用 `FileUserStore.register()` 注册用户
   - `002`：验证密码，成功后加入 `OnlineUsers`
   - `003`：从 `OnlineUsers` 移除，关闭连接
   - `004`：返回在线用户列表
   - `005`：校验身份后将消息放入 `Queue`
5. 异常时断开连接并清理资源

## SendMessageService 消息转发线程

采用生产者-消费者模式：

1. 死循环从 `queue.Queue` 中 `get()` 取出消息
2. 解析消息的 `(srcId, desId, msg)`
3. 在 `OnlineUsers` 中查找目标用户的 Socket
4. 若目标在线，发送 `106,srcId,desId,msg`
5. 若目标不在线，向发送方发送 `ERR,desId,FAIL:目标用户不在线`

## OnlineUsers 在线用户管理

- 内部使用 `dict` 存储 `{用户名: Socket}` 映射
- 所有读写操作通过 `threading.Lock` 保护，确保线程安全
- 提供 `add()`、`remove()`、`get()`、`list_users()` 方法

## FileUserStore 用户持久化

- 读取 `data/users.json` 文件加载到内存 `dict`
- 注册新用户时更新内存并写入文件
- 抽象出 `UserStore` 接口，便于将来替换为数据库实现

# 概论

## 选题以及意义

学生管理是高校教务工作中最基础、最核心的环节之一。从学生信息的录入、修改、查询到删除，每一个操作都直接影响教务管理的效率与准确性。本项目选择"Java+Spring Boot 学生管理系统设计与实现"作为课程设计题目，旨在运用现代 Web 开发技术构建一个轻量级、易部署的学生信息管理系统。

  3. 系统校验必填项是否完整
  4. 前端通过 `fetch()` 发送 POST 请求到 `/api/students`
  5. 后端将学生数据保存到仓库并持久化到 JSON 文件
  6. 返回成功提示，刷新学生列表
- **后置条件**：新学生记录已保存，列表中显示新增记录

### 用例2：查询学生

- **参与者**：教务管理员
- **前置条件**：系统首页已加载
- **主流程**：
  1. **查看全部**：页面加载时自动请求 `GET /api/students`，展示所有学生列表
  2. **按学号搜索**：在搜索框输入学号，点击"搜索"按钮
  3. 前端发送 `GET /api/students/{studentId}` 请求
  4. 后端返回匹配的学生信息或 404 错误
- **后置条件**：表格显示查询结果

### 用例3：修改学生

- **参与者**：教务管理员
- **前置条件**：学生列表中已有记录
- **主流程**：
  1. 管理员点击某学生记录的"编辑"按钮
  2. 弹出编辑模态框，显示当前信息
  3. 修改姓名、性别、年龄、专业等字段
  4. 点击"保存"按钮
  5. 前端发送 `PUT /api/students/{studentId}` 请求
  6. 后端更新数据并持久化
  7. 关闭模态框，刷新列表
- **后置条件**：学生信息已更新

### 用例4：删除学生

- **参与者**：教务管理员
- **前置条件**：学生列表中已有记录
- **主流程**：
  1. 管理员点击某学生记录的"删除"按钮
  2. 弹出确认对话框
  3. 确认后，前端发送 `DELETE /api/students/{studentId}` 请求
  4. 后端删除记录并持久化
  5. 刷新列表
- **后置条件**：该学生记录已从列表和数据文件中移除

# 详细设计

## Application 主启动类

- 继承 `SpringBootServletInitializer`，支持 WAR 部署到外部 Tomcat
- `@SpringBootApplication` 注解开启自动配置
- `main()` 方法调用 `SpringApplication.run()` 启动内嵌 Tomcat

## Student 实体类

包含五个属性：

| 属性 | 类型 | 说明 |
|------|------|------|
| `studentId` | String | 学号，唯一标识 |
| `name` | String | 姓名 |
| `gender` | String | 性别 |
| `age` | Integer | 年龄 |
| `major` | String | 专业 |

Jackson 自动将 Student 对象序列化为 JSON 格式。

## StudentRepository 数据仓库

核心设计：

- 使用 `LinkedHashMap<String, Student>` 作为内存缓存，key 为学号
- 构造函数中从 `students.json` 文件加载数据（`loadData()`）
- 每次增删改操作后调用 `persistData()` 将全量数据写入文件
- `@Value("${app.data.file}")` 注入配置文件中的数据文件路径
- 预置三条测试数据：3123004163-张逸壕、3123004001-李明、3123004002-王芳

数据流程：

```
启动 → loadData() 从 students.json 读取 → 存入 LinkedHashMap
增删改 → 更新 LinkedHashMap → persistData() 写入 students.json
查询 → 直接从 LinkedHashMap 读取
```

## StudentRestController 控制器

- `@RestController` = `@Controller` + `@ResponseBody`
- `@RequestMapping("/api/students")` 类级别映射
- `@Autowired` 注入 `StudentRepository`
- GET/POST/PUT/DELETE 分别对应查、增、改、删操作
- 查询/修改/删除时若学生不存在，抛出 `StudentNotFoundException`（自动返回 404）

## 前端管理界面

### 界面布局

- **顶部栏**：显示系统名称、学号（3123004163）、姓名（张逸壕）
- **新增区域**：学号、姓名、性别、年龄、专业输入框 + "添加学生"按钮
- **列表区域**：搜索框 + "搜索"/"刷新列表"按钮 + 学生数据表格
- **编辑模态框**：修改姓名、性别、年龄、专业 + "保存"/"取消"按钮

### 前端交互流程

```
页面加载 → fetch GET /api/students → 渲染表格
点击"添加" → fetch POST /api/students → 刷新列表
点击"搜索" → fetch GET /api/students/{id} → 渲染单条结果
点击"编辑" → 打开模态框 → fetch PUT /api/students/{id} → 刷新列表
点击"删除" → confirm 确认 → fetch DELETE /api/students/{id} → 刷新列表
```

| 项目 | 要求 |

## 部署步骤

### 方式一：Eclipse 中部署到外部 Tomcat

1. 将项目导入 Eclipse（`Import` → `Maven` → `Existing Maven Projects`）

4. 浏览器访问 `http://localhost:8080/homework11-spring-boot/`

### 方式二：命令行运行（内嵌 Tomcat）

1. 进入项目目录
2. 执行 `mvn spring-boot:run`
3. 浏览器访问 `http://localhost:8080/`

## 数据文件

- 学生数据保存在项目根目录的 `students.json` 文件中
- 首次运行时自动创建并写入预置数据
- 后续增删改操作实时更新该文件

# 课程设计总结

通过本次课程设计，我获得了以下收获：

1. **Spring Boot 框架理解**：深入理解了 Spring Boot 的自动配置原理，与作业10中手动配置 Spring MVC 相比，Spring Boot 大大简化了项目搭建过程

2. **RESTful API 设计**：掌握了 REST 风格的接口设计，理解了 HTTP 方法与 CRUD 操作的对应关系，以及 JSON 数据格式的序列化/反序列化

3. **前后端分离开发**：实践了前端（HTML+JS+CSS）与后端（Spring Boot REST API）的分离式开发模式，前端通过 fetch 异步调用后端接口

4. **数据持久化设计**：实现了基于 JSON 文件的持久化方案，理解了内存缓存 + 文件读写的数据管理策略，为将来使用数据库（如 JPA + MySQL）打下了基础

5. **项目整体能力提升**：从需求分析到总体设计、详细设计、编码实现、测试验证，完成了一个完整的软件开发流程，提升了工程实践能力

在实际开发中遇到的主要挑战是前端与后端的联调——确保 fetch 请求的 URL 路径、请求方法、请求体格式与后端控制器完全匹配。通过浏览器开发者工具的 Network 面板调试，最终成功解决了前后端通信问题。
